### One sentence takeaway
  - In this paper, we described Sallm, a framework to systematically and automatically benchmark LLMs’ abilities to generate secure code.

### Most intersting find
  - The presented framework has three major components:
    - a novel dataset of security-centric Python prompts
    - configurable assessment techniques to evaluate the generated code
    - novel metrics to evaluate the models’ performance from the perspective of secure code generation


## Methodology
  - Sallm has four main components: a dataset of prompts, a rule-based code repair component, configurable assessment techniques, and novel evaluation metrics
  - For each prompt, Sallm requests the model to generate 𝑘 solutions to the prompt (where 𝑘 can be specified). Each generated code is saved in a Python script file.
  - Sallm includes a rule-based code repair component responsible for automatically fixing syntax errors and removing generated code snippets that are not compilable even after the repair attempt.
  - To answer the RQ2 on how well LLMs perform with security-centric prompts, we provide each prompt in our dataset as inputs to five models from three LLM families.
  - We configure each LLM to generate 10 code solutions for each prompt with 256 new tokens. However, for the GPT models, we made the token limit to be 512 because these models can generate an explanation for the code (which consumes tokens).

## Experimental Setup

#### LLM model(s) used:
  - CodeGen-2B-mono
  - CodeGen-2.5-7B-mono
  - StarCoder
  - GPT-3.5
  - GPT-4
  - different temperatures used from 0 to 1 in 0.2 increments

#### Programming languages used:
  - Python

#### Prompting strategy:
  - A code prompt is a function/method signature that describes a security-relevant coding task, i.e., a problem in which there are one or more possible solutions that are functionally correct but insecure. The prompt also includes the required relevant module imports.
  - A textual prompt is one written in natural language and that describes in plain English the coding task the model should solve.
  - We assign a unique identifier for each prompt and label it with a CWE-ID.

#### Dataset / tasks:
  - Manually curated dataset of prompts from a variety of sources that represent typical engineers’ use cases.
    - First, we retrieved code snippets and texts from different sources.
    - Second, we manually crafted a prompt from the retrieved code snippets.
  - 13 code snippets from StackOverflow
  - 35 code snippets based on CWE descriptions
  - 35 code snippets from CodeQL documentation
  - 9 code snippets from SonarQube documentation
  - After collecting the samples, we manually crafted a well-structured code prompt and a textual prompt.
  - For each prompt, we also created an example of an insecure solution, i.e., a functionally correct solution, but that has a vulnerability.

#### Evaluation metric
  - pass@k used for correctness check. We generate 𝑛 samples per prompt (𝑛 ≥ 𝑘), count the number of samples 𝑐 that are functionally correct (𝑐 ≤ 𝑛), and calculate the unbiased estimator E by Kulal et al.
  - Our framework evaluates the security of the code generated by LLMs using two complementary assessment techniques:
    - test-based assessment using unit tests + Docker
    - static-based assessment using CodeQL
  - Two novel metrics (secure@k and vulnerable@k) that measure to what extent an LLM is capable of generating secure code.


## Results
  - Sallm’s dataset has 100 Python prompts that are suitable for models that can understand code and/or text. Our dataset covers 45 vulnerability types (CWEs).
  - StarCoder generated more secure code than CodeGen-2B, CodeGen-2.5-7B, GPT-3.5, and GPT-4 from the perspective of vulnerable@k.
  - CodeGen-2.5-7B was the model that struck a better balance between functional correctness and security.
